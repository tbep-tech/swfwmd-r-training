1.3 Goals and framing

R is a language for statistical computing and general purpose programming. It is one of the best languages for data science and analysis.

This training is not intended to teach R from the ground up. Instead, it is meant for participants who can already run code in RStudio and want to become more effective using R for day-to-day work. The emphasis is on reproducible workflows, understanding inherited scripts, troubleshooting errors, and building confidence with realistic analysis tasks.

You should be able to answer these questions at the end of this session:

  • How should I organize an R project so my work is reproducible?
  • How can I read and understand an R script that somebody else wrote?
  • How do relative file paths and project structure prevent common errors?
  • How can I bring external data into a project and inspect it quickly?
  • Where can I go for help?
  • How can AI help me troubleshoot without replacing the learning process?

1.4 Why this training is different

The original CERF workshop that this repository was modeled on was designed for beginners. This workshop has a different audience. Participants already have some experience with R and will likely be using scripts they received from colleagues, shared examples, or past projects. The key challenge is often not writing code from scratch, but figuring out:

  • what a script is trying to do,
  • why it fails,
  • how to adapt it safely, and
  • how to organize work so those problems are easier to solve next time.

For that reason, this opening lesson focuses on workflow and troubleshooting rather than basic syntax.

1.5 Project-oriented workflow in RStudio

RStudio remains the primary teaching environment for this workshop. It gives you one place to write scripts, run code, inspect objects, review files, and view plots. For professional work, the most important habit is not simply knowing where the panes are, but using RStudio in a way that makes your projects portable and understandable.

1.5.1 Why projects matter

An RStudio Project gives a single home for data, scripts, figures, and outputs associated with one analysis. This matters because many frustrating R problems are really project management problems:

  • file not found errors from incorrect paths,
  • code that only works on one person’s computer,
  • overwritten outputs,
  • and difficulty understanding where a script expects its inputs to live.

When you work inside a project, your scripts can use relative paths such as data/dat.csv instead of hard-coded absolute paths tied to one machine.

There are four panes in RStudio:

  • Source: Your primary window for writing code to send to the console, this is where you write and save R “scripts” for your code
  • Console: This is where code from your scripts is executed in R - you will typically not write code here
  • Environment, History, etc.: A tabbed window showing your working environment, code execution history, and other useful things
  • Files, plots, etc.: A tabbed window showing a file explorer, a plot window, list of installed packages, help files, and viewer

1.5.2 RStudio projects

It is good practice to start each new task in a fresh project. The project directory becomes the working context for your analysis and should usually contain subfolders for raw data, scripts, derived outputs, and figures.

To create a new project, click on the File menu at the top left and select ‘New project…’

Now we can use this project for our data and any scripts we create.

1.5.3 Relative paths and folder structure

Inside a project, your script should point to files relative to the project root:

# project-relative file paths
data_file <- file.path("data", "dat.csv")
station_file <- file.path("data", "statloc.csv")

This is preferable to absolute paths such as C:/Users/name/Desktop/..., which usually fail as soon as the code moves to a different computer or folder.

1.5.4 Scripting

In most cases, you should write code in scripts and send it to the console as needed. A saved script is easier to share, troubleshoot, review, and reuse than code typed interactively into the console and lost when the session closes.

Open a new script from the File menu:

After you write code in your script, use the Run button or ctrl+enter (cmd+enter on a Mac) to send lines to the console.

1.7 Working with scripts from colleagues

Many participants in this workshop already use R by modifying scripts they received from others. That is a realistic and productive way to work, but it also creates a common failure mode: a script appears opaque until you know how to read it.

A good first pass through an inherited script is to answer four questions:

  1. What packages are loaded?
  2. What files are imported?
  3. What objects are created or modified?
  4. What outputs are produced?

The point is not to understand every detail at once. The point is to locate the major stages of the workflow and identify where the code depends on your local setup.

1.7.1 A simple review workflow

library(tidyverse)

data_file <- file.path("data", "dat.csv")
station_file <- file.path("data", "statloc.csv")

dat <- read_csv(data_file)
statloc <- read_csv(station_file)

glimpse(dat)

The combination of file.path(), read_csv(), and glimpse() gets you surprisingly far when you are trying to understand what a script expects and what kind of data it is using.

1.8 Accessing SWFWMD data

One goal for this workshop is to use examples that look more like real SWFWMD work. A useful way to do that is to access data from the SWFWMD data portal and then use those data in later wrangling, plotting, and spatial examples.

For now, think of portal access as another data import step. The main ideas are:

  • build a request URL carefully,
  • import the result into R,
  • inspect the structure immediately,
  • and save or reuse the result in later steps of the workflow.

An example URL pattern for requesting station metadata looks like this:

station_url <- paste0(
  "https://edp.swfwmd.state.fl.us/KiWIS/KiWIS?",
  "service=kisters&type=queryServices&request=getStationList",
  "&station_no=*&datasource=0",
  "&returnfields=station_no,station_name,station_latitude,station_longitude",
  "&format=csv&csvdiv=,"
)

station_url

Later lessons can use a finalized portal example and a selected SWFWMD dataset as an anchor workflow for the rest of the training.

1.10 Troubleshooting and getting help

Troubleshooting is a core skill for this workshop. Before changing code, make the problem smaller and more specific. Useful first checks include:

  • Does the file path exist?
  • Did the package load successfully?
  • What is the class or structure of the object?
  • Which line produced the error?
  • Is the code failing because the input data are different than expected?

Many problems become much easier once you inspect the object directly instead of guessing what it contains.

1.10.1 Help from the console and documentation

Getting help from the console is straightforward:

?read_csv
help(package = "dplyr")
??mutate

You should also get comfortable with package websites, vignettes, and worked examples. Searching the exact error message is often more effective than searching a vague description of the problem.

1.11 Generative AI as a support tool

Generative AI can be genuinely useful for this audience, especially when you need help understanding an inherited script or unpacking an error message. In many cases it is faster than a web search for:

  • explaining what a script is doing step by step,
  • suggesting why an error occurred,
  • translating unfamiliar code into plainer language,
  • or outlining a debugging strategy.

The tradeoff is that AI can sound confident while being wrong, can overcomplicate a fix, and can nudge users toward copying code they do not understand. Use it to support your reasoning, not replace it.

Some practical guidelines:

  • Give the model the code, the error message, and a short description of what you expected to happen.
  • Ask it to explain before asking it to rewrite.
  • Verify every suggested fix.
  • Never paste sensitive or proprietary information into a public tool.

1.12 Exercise 3

Use a short script from this lesson or one of your own and practice a structured review.

  1. Identify which packages it loads.

  2. Identify which files it expects.

  3. Identify the main objects it creates.

  4. If something fails, write down the exact error and list the first two things you would inspect before changing the code.

1.13 Summary

In this lesson we reframed the workshop around reproducible workflow rather than basic syntax. The most important themes were: use projects, keep paths relative, inspect data early, read scripts in stages, and troubleshoot carefully before changing code. Next, we can build on that foundation with data wrangling tasks that move from clear toy examples into SWFWMD-style workflows.
